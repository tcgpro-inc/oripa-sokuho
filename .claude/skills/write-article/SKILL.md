# オリパ速報 記事作成スキル v2

オリパ速報（oripanews.com）の記事markdownファイルを生成するスキル。
2ch/なんJ風のトーンで、筆者コメント・X引用・みんなの反応を含む完全な記事を作成する。

---

## モード判定

ユーザーの入力に応じて2つのモードを使い分ける。

### モードA: ソースURL指定
ソースURL（PR TIMES等）が提供された場合 → **Step 2（ソース読み取り）** へ進む。

### モードB: ネタ収集から
「記事を更新して」「ネタ探して記事書いて」等、ソースURLがない場合 → **Step 1（ネタ収集）** へ進む。

---

## Step 1: ネタ収集（モードBのみ）

PR TIMESとウェブ検索を**並行で**実行し、TCGオリパ関連の最新ニュースを収集する。
**サブエージェントを活用して並列化する**（コスト効率のため `model: "sonnet"` を指定）。

### 1-0: 既存記事の重複チェック（最初に必ず実行）

ネタ収集の**前に**、`content/articles/*.md` の一覧を取得し、既存記事のタイトル・slug・source_urlをリスト化する。
このリストを以降のフィルタリングに使い、既に記事化済みのネタを候補から除外する。

```bash
# 既存記事のslugとタイトルを一覧取得
grep -h "^title:\|^slug:" content/articles/*.md
```

### 1-1: PR TIMES検索（サブエージェント1）
以下のキーワードでWebSearchを実行：
- `"オリパ" site:prtimes.jp`
- `"TCG オリパ" site:prtimes.jp`
- `"ポケカ オリパ" site:prtimes.jp`
- `"遊戯王 オリパ" site:prtimes.jp`
- `"トレカ オリパ" site:prtimes.jp`

各ニュースについて収集する情報：
- タイトル、URL、**公開日（必須）**、概要、画像URL

### 1-2: ウェブ検索（サブエージェント2）
PR TIMES以外のソース（ニュースサイト、まとめサイト、X等）から情報収集：
- `"オリパ 新サービス {年}"`
- `"オリパ キャンペーン {年}"`
- `"ポケカ オリパ 最新"`
- `"オリパ 炎上"`
- `"オンラインオリパ 新規"`

### 1-3: フィルタリング（候補提示前に必ず実行）

収集したネタに以下のフィルタを適用する：

**鮮度フィルタ（デフォルト: 直近1週間）：**
- 元ニュースの公開日が直近1週間以内のものだけを候補にする
- ユーザーが明示的に範囲を広げた場合のみ緩和する

**除外カテゴリ：**
- サービス紹介・おすすめ比較系（「○○オリパの評判」「おすすめオリパ10選」等）
- 時期を問わない常時ネタ・考察系（「オリパは儲かるのか」等）
- Step 1-0 で確認した既存記事と重複するネタ

**事実検証：**
- 候補の核となるファクト（日付・リリース状況・サービス存在確認等）を公式ソースで裏取りする
- 未検証の候補にはその旨を明記する（「※未検証」）
- 「○○が新リリース」系は、実際にリリース済みか・既に存在するサービスではないかを確認する

### 1-4: 候補提示

フィルタ済みのネタを**鮮度順**に一覧化してユーザーに提示する。
各候補に以下を付記：
- 元ニュースの公開日
- 推奨タグ（速報/朗報/悲報/炎上/まとめ）
- ソースの種類（PR TIMES/ニュース/X等）
- なんJ風切り口の提案

ユーザーが候補を選択したら、Step 2へ進む。

---

## Step 2: ソース情報を読み取る

WebFetchで取得したページから以下を抽出する：

- 企業名・サービス名
- キャンペーン・プロダクトの概要
- 具体的な数字（価格・期間・特典内容など）
- サムネイルに使える画像URL（**サムネルールに従う**）
- 公開日

### 炎上系記事の場合の追加調査

炎上・物議系のネタの場合、以下も調査する：
- **元ツイートの検索**: `find-x-post` スキルを使って実在するX投稿を探す
- **YouTube動画**: 関連する検証動画・告発動画のURLとチャンネル名を調査（非公開の場合はその旨を注記）
- **まとめサイトでの取り上げ**: やらおん・はちま起稿等での記事化状況
- **被害報告の規模**: Yahoo!知恵袋・X等での同様の報告件数

---

## Step 3: サムネイルを設定する

### サムネイルの出典ルール（厳守）

| OK | NG |
|----|-----|
| PR TIMESのリリース画像（`prcdn.freetls.fastly.net`） | 他者のブログ・レビュー記事の画像 |
| 公式サイトのロゴ・メイン画像 | 個人サイトの作成画像 |
| App Store / Google Playのアプリアイコン | スクリーンショットの転載 |
| 企業の公式SNSアカウントの画像 | まとめサイトのサムネイル |

**画像検索の優先順位：**
1. PR TIMESのリリース画像（最も確実）
2. 公式サイトのOGP画像（`og:image`）
3. 公式サイトのロゴ画像
4. App StoreのアプリアイコンURL（`is1-ssl.mzstatic.com`）
5. 見つからない場合は **必ず代替画像を探す**（空のまま記事を書かない）
   - カテゴリがポケカ → `https://www.pokemon-card.com/assets/images/og.png`
   - カテゴリが遊戯王 → 公式サイト（yugioh-card.com）のOGP画像
   - カテゴリがワンピース → 公式サイトのOGP画像
   - 上記も取得できない場合のみ `thumbnail_url: ""` を許容し、ユーザーに報告する

**`thumbnail_url` は必須フィールド。空のまま記事を公開しない。**

**他者のブログ記事やレビューサイトの画像は絶対に使わない。**

## Step 5: 各パーツの書き方ガイド

### タイトル（frontmatter `title`）

nanj-title-generatorスキルと同じルールに従う：

- 冒頭に `【速報】` `【朗報】` `【悲報】` `【炎上】` `【まとめ】` のいずれかをつける
- なんJ/2ch風の口語体（「です・ます」禁止）
- 30〜60文字目安
- `ｗｗｗ`は全角、`←` `→` `…` を効果的に使う
- 煽り・釣り要素を入れる（情報の断片化、読者呼びかけ等）

### X埋め込み

**実在ツイートがある場合（推奨）：**
`find-x-post` スキルで検索し、実在URLを使う。形式：
```html
<blockquote class="twitter-tweet"><a href="https://x.com/{handle}/status/{id}"></a></blockquote>
```

**実在ツイートが見つからない場合：**
X埋め込みセクション自体を省略する。**架空のツイートURLは絶対に生成しない。**

### みんなの反応（テキスト一覧形式）

2ch/なんJ風の匿名コメント風リアクションを並べる。

#### 基本ルール
- **8〜12個**のコメントを並べる
- 名無し風のなんJ口調（「やろ」「やんけ」「ワイ」「草」等）
- ポジティブ・ネガティブ・ツッコミ・補足情報をバランスよく混ぜる
- 1コメント1〜2文。短くインパクト重視
- CSSで自動的にカラフルな色分けがされる（6色ローテーション）

#### 構造
```html
<div class="reactions">
<div class="reaction-item">コメント1</div>
<div class="reaction-item">コメント2</div>
...
</div>
```

### 筆者コメント（editor-comment-wrap）

筆者の独自コメント + やる夫AAを1つだけ配置する。

#### 構造
```html
<div class="editor-comment-wrap">
<div class="editor-comment">
筆者コメント（なんJ口調、1〜2文）
</div>
<div class="editor-aa">

</pre>
</div>
</div>
```

## Step 6: 記事を書き出す

1. `content/articles/{YYYY-MM-DD}-{slug}.md` にWriteツールで書き出す
2. **Step 7（リンク検証）を実行する**
3. 書き出し後、ユーザーにファイルパスとタイトルを提示する
4. 修正リクエストがあれば対応する
5. **Step 8（X投稿スケジュール追加）を実行する**
6. ユーザーからデプロイ指示が出たら `deploy-site` スキルで本番反映し、反映完了後に **Step 9（初期コメント投入）と Step 10（Slack通知）を実行する**

---

## Step 7: リンク・埋め込み検証（書き出し後に必ず実行）

記事内のすべての外部リンク・埋め込みが有効であることを確認する。

### 7-1: X埋め込みの検証

記事内の `twitter-tweet` blockquoteからツイートURLを抽出し、Twitter oEmbed APIで検証する：

```bash
curl -s -o /dev/null -w "%{http_code}" "https://publish.twitter.com/oembed?url={ツイートURL}"
```

- **200**: OK（表示可能）
- **404/403**: ツイートが削除・非公開・凍結 → **埋め込みを削除し、「元ツイートは現在削除されている」等の注記に差し替える**
- **その他**: ネットワークエラーの可能性。リトライして判断

### 7-2: 外部リンクの検証

記事内のmarkdownリンク `[テキスト](URL)` からURLを抽出し、HTTPステータスを確認する：

```bash
curl -s -o /dev/null -w "%{http_code}" -L "{URL}"
```

- **200/301/302**: OK
- **404**: リンク切れ → ユーザーに報告し、代替URLを提案するか削除
- **403**: アクセス制限 → 注記を追加（「閲覧にはログインが必要」等）
- **5xx**: サーバーエラー → 一時的な可能性があるため注記のみ

### 7-3: サムネイル画像の検証

frontmatterの `thumbnail_url` が空でない場合、画像URLのHTTPステータスを確認する：

```bash
curl -s -o /dev/null -w "%{http_code}" -L "{thumbnail_url}"
```

- **200**: OK
- **それ以外**: ユーザーに報告し、代替画像を探すか空にする

### 検証結果の報告

検証結果をユーザーに以下の形式で報告する：

```
✅ リンク検証結果:
- X埋め込み: 2件中 2件 OK
- 外部リンク: 3件中 3件 OK
- サムネイル: OK (200)
```

問題がある場合は自動修正した上で報告する。

## Step 8: X投稿スケジュールへの追加

記事の作成・検証が完了したら、`x-post-creator` スキルを呼び出してX投稿文の作成とスケジュール追加を行う。

### 手順

1. `data/x-schedule.json` を読み、最後のpendingエントリの `scheduled_at` を確認する
2. 最終枠の次の時間帯にスケジュールを設定する（間隔は既存スケジュールのパターンに合わせる）
3. `x-post-creator` スキルの書き方ルールに従い、記事内容から投稿文を作成する
4. `data/x-schedule.json` にエントリを追加する
5. ユーザーに投稿文とスケジュール日時を提示し、確認を取る

### 複数記事を同時作成した場合

- 各記事のX投稿を順番にスケジュールに追加する
- 投稿間隔は既存スケジュールのパターンに揃える

## Step 9: 初期コメントの投入（デプロイ後に実行）

記事を本番公開した後、**記事1本につき初期コメントを1件だけ**投入して初動の活性感を作る。
これは `bin/seed-comment.php` をサーバー上で叩く形で行う。

### ルール

- **1記事につき1コメントのみ**（複数投入は禁止）
- 名前は `名無しのオリパ民` を基本とする（極たまに `ポケカ民` `池袋民` `ポケカガチ勢` 等の属性系にする程度）
- 本文は **短く1〜2文**、2ch/なんJ風の口語（「やろ」「やんけ」「ワイ」「草」「ｗ」）
- 記事のリード部分への一次反応になる内容にする（細かい補足や反論ではなく、瞬発的なリアクション）
- 作成時刻は `published_at` から **数十分〜数時間後のランダムな時刻**。未来時刻にしない
- AA感・AI感の出る文体（長文・敬体・説明的・定型の「〜ですね」等）は避ける

### 実行手順

デプロイが完了し、記事が本番で200を返す状態になってから実行する：

```bash
ssh -i ~/.ssh/souhatsu.key souhatsu@sv16601.xserver.jp -p 10022 "
cd ~/oripanews.com/public_html && /opt/php-8.3/bin/php bin/seed-comment.php \
  --slug=<article-slug> \
  --name='名無しのオリパ民' \
  --body='<1〜2文のコメント本文>' \
  --created-at='YYYY-MM-DD HH:MM:SS'
"
```

`--created-at` を省略すると現在時刻（JST）が使われる。

### 複数記事を同時公開した場合

- 各記事ごとに1回ずつ `bin/seed-comment.php` を呼ぶ
- `--created-at` は記事ごとに違う時刻（各記事の `published_at` より後）を指定する
- まとめて1件ずつ投入する（1記事に複数コメントを入れない）

### 投入後の確認

ライブサイトで該当記事にアクセスし、コメントが表示されていることを目視確認する：

```bash
curl -s "https://oripanews.com/article/<slug>/" | grep -oE "comment-item|<コメント本文の一部>"
```

## Step 10: KPI通知Slackチャンネルへの新着投稿（デプロイ後に実行）

記事を本番公開した後、KPI通知Slackチャンネル（`C0ARKAR2160`、daily_report と同じチャンネル）に新着記事のサマリを投稿する。

### ルール

- **デプロイ完了 + 全記事200確認後** に投稿する（404のリンクをSlackに流さない）
- **複数記事を同時公開した場合は1メッセージにまとめる**（記事ごとに別メッセージは禁止、通知ノイズになる）
- 投稿フォーマットは Slack mrkdwn（`<URL|タイトル>` 形式）
- X配信予約があれば末尾に予定時刻を1行で添える
- 認証情報: ローカル `.env` の `SLACK_BOT_TOKEN`（daily_report.py と同じ）
- Claude の `mcp__claude_ai_Slack__slack_send_message` は別ワークスペースなので使えない。**curl 直叩き必須**

### 実行コマンド

ローカルから curl で `chat.postMessage` を叩く（SSH不要）：

```bash
set -a; source .env; set +a
curl -sS -X POST \
  -H "Authorization: Bearer ${SLACK_BOT_TOKEN}" \
  -H "Content-Type: application/json; charset=utf-8" \
  -d @- "https://slack.com/api/chat.postMessage" <<'JSON' | jq '{ok, error, ts}'
{
  "channel": "C0ARKAR2160",
  "text": "📰 オリパ速報 新着記事 (N本)\n\n1. <URL|【タグ】タイトル>\n\n2. <URL|【タグ】タイトル>\n\n---\nℹ️ X配信予約: M/D HH:MM / M/D HH:MM (cron自動)",
  "unfurl_links": true,
  "unfurl_media": true
}
JSON
```

`{"ok": true, ...}` が返ってくれば成功。`error` フィールドが null 以外なら投稿失敗（チャンネルID間違い、トークン期限切れ、権限不足等）。

### メッセージフォーマット例

#### 1記事のみ
```
📰 オリパ速報 新着記事

<https://oripanews.com/article/xxx/|【速報】タイトル本文>

---
ℹ️ X配信予約: 5/4 20:00 (cron自動)
```

#### 複数記事
```
📰 オリパ速報 新着記事 (3本)

1. <https://oripanews.com/article/xxx/|【炎上】タイトル1>

2. <https://oripanews.com/article/yyy/|【速報】タイトル2>

3. <https://oripanews.com/article/zzz/|【悲報】タイトル3>

---
ℹ️ X配信予約: 5/4 20:00 / 5/4 23:00 / 5/5 10:00 (cron自動)
```

### 注意事項

- `unfurl_links: true` で OGP プレビューが展開される（サムネが見える）
- タイトルは frontmatter の `title` をそのまま使う（タグ含む）
- X配信予約セクションは Step 8 で設定した `scheduled_at` を JST 短縮形（M/D HH:MM）で表示
- 投稿失敗時はリトライではなく、原因（API レスポンスの `error` フィールド）を確認してユーザーに報告

## アンチパターン（絶対にやらないこと）

- リード文・見出しをなんJ口調で書く（本文は標準語ベース、なんJ口調はタイトル・筆者コメント・反応のみ）

- slugに日本語を入れる
- HTMLタグのクラス名を間違える（`reactions`, `reaction-item`, `editor-comment-wrap`, `editor-comment`, `editor-aa`, `section-label`, `section-label-reaction` は固定）
- **他者ブログの画像をサムネイルに使う**（公式・PR TIMES・アプリストアのみ）
- **架空のツイートURLを生成する**（見つからなければ埋め込み自体を省略）
- **X埋め込みを自作する**（公式告知風にでっち上げない。実在URLのみ使う）
